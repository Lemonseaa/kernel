# CheckpointAI 下一阶段规划：Autonomous Loop

> 讨论稿：这份文档保留 Hermes 讨论过程中的原始想法。正式 V7 口径以 `docs/V7_PLAN.md` 为准。
> 已收紧的关键边界：V7 不做无人值守上线，不把 CrewAI 作为核心依赖，不允许 Agent 直接执行改动。

## 背景

CheckpointAI V1-V6已完成。当前系统能记录和优化AI输出，但还不够"自动化"——proposal仍需人想，系统只是验证。

下一阶段目标：让系统能自动发现问题、分析原因、生成建议、执行验证，形成真正的优化闭环。

---

## 核心理念

```
1. 高速自动化
   - 99%的proposal自动执行
   - 可随时回滚
   - 人只做很小部分的审批

2. Hermes是可选的对话界面
   - 没有Hermes系统也能跑
   - Hermes只是增强体验
   - 人通过Hermes了解进度、配置系统

3. 每个业务线一个内置Agent系统
   - 业务线之间上下文隔离
   - 内置系统协调多个外置Agent

4. 渐进式自动化
   - 先验证闭环能跑
   - 再逐步增加自动化程度
```

## 系统架构

### 整体布局

```
业务层 → Workflow可视化
              ↓
        [锁定] [分支] [查看历史]

右悬浮：Hermes（可选对话）

三层结构：
- 看板层：Dashboard/Runs/Reports等
- 业务层：业务分区/Workflow/构建Agent
- 审批层：Approval Inbox
```

### 内置 vs 外置Agent

```
内置Agent系统：
- 在CheckpointAI内部
- 负责：协调外置Agent、优化闭环

外置Agent系统：
- TradingAgents / ContentAgents / CustomerAgents等
- 在CheckpointAI外部
- 负责：实际执行任务
```

### 每个业务线一个内置Agent系统

```
业务线A：
  内置Agent系统A
    ↓ 协调
  外置Agent A1, A2, A3

业务线B：
  内置Agent系统B
    ↓ 协调
  外置Agent B1, B2

业务线之间完全隔离
```

## 内置Agent架构

### 4个Agent

```
1. Investor（调查者）
   = 主动探索 + 监控异常
   - 发现数据异常
   - 分析可能原因
   - LLM：小型快速模型

2. Proposer（建议者）
   - 分析原因
   - 生成proposal
   - 评估风险
   - LLM：大型推理模型

3. Refiner（优化者）
   - 执行proposal
   - 验证效果
   - 更新知识库
   - LLM：中型稳定模型

4. Safety Agent（安全员）
   - 监控执行
   - 发现异常触发回滚
   - 记录失败模式
   - LLM：小型快速模型
```

### CrewAI实现

```
使用CrewAI的：
- Agent定义
- Task定义
- Crew编排
- 共享上下文

CrewAI和Hermes独立：
- CrewAI是后台自动化引擎
- Hermes是前端对话界面
```

## 自动循环流程

```
1. Investor持续探索
   ↓ 发现异常/机会
2. Proposer分析原因
   ↓ 生成proposal
3. 风险评估
   ├── 低风险 → 自动执行 → Refiner验证
   └── 高风险 → 人审批 → Refiner验证
   ↓
4. Refiner验证效果
   ↓
5. 更新知识库
   ↓
6. 回到1（持续循环）
```

### 改动策略

```
不冲突的改动 → 一起执行
冲突的改动 → 人来决定

一次只做一个改动
改动尽量小
```

---

## 风险分级

```
低风险（自动执行）：
- 小参数微调
- 经过多次验证的改动
- 预期效果明确

中风险（通知确认）：
- 中等幅度改动
- Hermes通知人，人说"知道了"

高风险（必须审批）：
- 大幅改动
- 核心逻辑改变
- 涉及生产环境
```

---

## Safety Agent回滚机制

```
自动回滚条件：
- 执行后指标立即恶化
- 验证失败
- 异常错误

回滚流程：
Safety Agent检测 → 自动回滚 → 记录原因 → Hermes通知人
```

## 动态平衡机制

### 需要解决的问题

```
1. 过拟合
   - 系统一直改参数，改到完美匹配历史数据

2. 震荡
   - A→B→A→B来回改

3. 梯度崩溃
   - 一直往一个方向优化，到局部最优就停

4. 负向优化
   - 改完后整体效果变差
```

### 需要的平衡机制

```
1. 约束机制（Guardrails）
   - 参数范围限制
   - 改动幅度限制
   - 禁止改动项

2. 冷却机制（Cooldown）
   - 同一参数改动后等待N次验证
   - 防止震荡

3. 多样性保护（Diversity）
   - 强制生成多样化proposal
   - 避免局部最优

4. 回归检测（Regression Detection）
   - 持续监控改动后效果
   - 及时回滚

5. 预算控制（Budget Control）
   - 每天最多N次优化尝试
   - 控制优化频率
```

## 版本分支机制

### 人的操作

```
1. 锁定（Lock）
   - 把当前配置打tag
   - Agent不能自动改这个参数

2. 分支（Branch）
   - 基于锁定版本开新分支
   - 在新分支上继续优化

3. 切换（Switch）
   - 切换到某个分支作为主版本

4. 回滚（Rollback）
   - 切回之前的锁定版本
```

### UI vs Hermes

```
UI按钮（直接）：
[锁定当前配置]
[创建新分支]
[切换版本]
[回滚]

Hermes（对话）：
"帮我锁定这个配置"
"开一个新分支"
```

↓

外置Agent执行结果 → 统一格式 → UI可查看

## 实施计划

### V7.1 Alpha：第一阶段

```
目标：
- 3个Agent能跑起来
- 对接一个外置系统
- 验证闭环能工作

执行顺序：
1. 实现3个Agent（Investor/Proposer/Refiner）
2. 实现Safety Agent
3. 对接第一个外置系统
4. 验证闭环
5. 扩展到更多外置系统
```

### 渐进自动化

```
第一阶段：
- 全部proposal人审批
- 系统只是记录和验证

第二阶段：
- 低风险自动执行
- 高风险人审批

第三阶段：
- 根据置信度自动执行
- 持续学习改进
```

### 验证指标

```
- 优化次数/天
- 改动成功率
- 平均效果提升
- 回滚次数

如果指标没改善 → 说明系统在乱改 → 调整约束
```
